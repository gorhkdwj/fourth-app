## 1. 배경 / 문제

한 카페가 4개 경로(앱리뷰·설문·전화메모·인스타DM)로 받은 고객 피드백이 **한 파일에 뒤섞여** 있다.
- 받은날짜가 **4가지 형식**으로 섞임: `2026-05-02` · `2026/05/03` · `5월 4일` · `26.5.11`
- **별점이 빈 칸인 행이 많음** (22행 중 절반 이상)

사장님의 요청은 **딱 하나**:
> "지금 **가장 급한 불만**이 뭔지 **한 화면**에서 보고 싶다."

## 2. 목표 / 비목표

### 목표 (이번에 만든다)
- **G1.** 받은날짜를 `YYYY-MM-DD` 한 형식으로 통일한 정리본 CSV
- **G2.** 각 행에 **유형**(불만/요청/칭찬/문의)과 **감정**(긍정/부정/중립) 열을 추가한 분류본 CSV — *분류는 앱이 아니라 Claude가 직접 수행*
- **G3.** 유형별 개수 + **'가장 급한 불만 TOP 3'** 를 보여주는 **Streamlit 한 화면**

### 비목표 (의도적으로 안 만든다 = 덜어내기)
- 경로별/날짜별 추세 그래프, 워드클라우드, 필터 UI 등 "있으면 좋은" 시각화
- 앱이 런타임에 Claude API를 호출하는 구조 (분류는 사전 1회, 결과는 정적 CSV)
- 로그인/계정/멀티유저
- (선택과제) Supabase 저장·배포·Skill화 — 필수가 돌아간 뒤에만 검토

## 3. 핵심 정의 — '가장 급한 불만' (과제의 머리 쓸 포인트)

별점 1점만 급한 게 아니다. **무엇이 깨졌나(심각도) × 고객이 얼마나 불만인가(강도)** 의 곱으로 정의한다.
이는 CS·VoC 업계(Zendesk Intelligent Triage, IrisAgent, edesk 류)가 우선순위를 매기는 방식과 같다 —
**별점이 아니라 카테고리(인텐트)·키워드·감정강도**가 우선순위를 결정한다.

```
긴급도 U = 심각도 S × 불만강도 I      (유형 = 불만일 때만, 그 외 0)
불만강도 I = max(별점신호, 텍스트신호)   ← 별점은 제외되지 않고 한 입력으로 참여
```

**① 심각도 S** — SERVQUAL 서비스품질 차원 중요도(신뢰성 최우선) 근거:

| 차원 | S | 해당 |
|---|---|---|
| 신뢰성(Reliability) | 4 | 결제·포인트·환불·주문정확성·앱오류 |
| 응답성(Responsiveness) | 3 | 과도한 대기·진동벨 미작동 |
| 유형성·가치(Tangibles) | 2 | 좌석·와이파이·화장실·가격 |
| 제품 기호 | 1 | 단맛 등 단순 취향 |

**② 불만강도 I = max(별점신호, 텍스트신호)** — 두 신호 중 '더 위험한 쪽' 채택('급한 불만을 놓치지 않기'):

| 신호 | 매핑 |
|---|---|
| 별점신호 (있을 때만) | 1→3, 2→2, 3→1, 4·5→0 |
| 텍스트신호 | 긴급/금전 키워드(환불·결제·포인트·오류·반복) 또는 강한 부정 → 3 / 일반 부정 → 2 / 중립 → 1 / 긍정 → 0 |

- **불만(유형)만 점수화**. 칭찬/문의/요청은 0.
- 동점이면 **금전 손실·환불 요구 우선**(재무·법적 리스크) → 최근 날짜 순.
- 심각도 가중치·키워드 목록은 코드 상수로 분리해 **언제든 조정 가능**.
- **근거 출처**: SERVQUAL(Parasuraman·Zeithaml·Berry 1988); 결측 별점 상수 임퓨테이션의 위험은
  Chakraborty·Kim·Sudhir(2022, *Journal of Marketing Research*)가 지적 → 텍스트 기반 추론으로 대체.

**왜 곱셈인가**: 리스크 매트릭스(영향도×심각도) 논리. 둘 다 높은 좌상단만 진짜 급한 불만으로 떠오른다.

## 4. 데이터 파이프라인 (① → ② → ③, 한 단계씩)

```
원본 CSV
  │  ① 날짜 통일 (코드 규칙: 4형식 → YYYY-MM-DD, 연도 없으면 2026 가정)
  ▼
feedback_step1_dates.csv
  │  ② Claude 분류 (유형·감정 + 긴급도점수) ※ Claude가 직접, Few-shot으로 기준 고정
  ▼
feedback_classified.csv   ← 대시보드의 단일 입력
  │  ③ Streamlit이 읽어서 표시 (계산만, 분류 안 함)
  ▼
한 화면 대시보드
```

### ② 분류 스키마
- `유형` ∈ {불만, 요청, 칭찬, 문의}
- `감정` ∈ {긍정, 부정, 중립} — 별점 있으면 별점 우선 참고, 없으면 내용으로 추정
- `긴급도` = 위 3장 공식으로 산출한 정수
- `급한이유` = 한 줄 근거 (TOP 3 카드에 표시)

### 분류 규칙 + Few-shot 예시 (분류 흔들림 방지)
**주 유형 판정**: 불만족·문제 지적이 있으면 개선 제안이 섞여 있어도 **불만**, 순수 질문은 **문의**, 불만족 없는 순수 제안만 **요청**.
- "결제는 됐는데 포인트가 안 쌓였어요. 환불해주세요." → 유형=불만, 감정=부정 (금전 사고)
- "신메뉴 라떼 너무 달아요. 당도 조절 옵션 있으면 좋겠어요." → 유형=불만, 감정=부정 (불만+제안 → 불만 우선)
- "시즌 한정 메뉴 자주 나왔으면 좋겠어요." → 유형=요청, 감정=긍정 (불만족 없는 순수 제안)
- "케이크 퀄리티가 베이커리급이에요. 강력 추천!" → 유형=칭찬, 감정=긍정
- "단체 예약도 가능한가요?" → 유형=문의, 감정=중립

## 5. 화면 설계 (한 화면, 덜어내기)

```
┌──────────────────────────────────────────────┐
│  ☕ 카페 VoC — 지금 가장 급한 불만            │
│  기간 2026-05-02 ~ 05-23 · 피드백 22건         │
├──────────────────────────────────────────────┤
│  [불만 N] [요청 N] [칭찬 N] [문의 N]  ← 4 지표 │
├──────────────────────────────────────────────┤
│  🔥 가장 급한 불만 TOP 3                        │
│   1. (별점·경로·날짜)  내용 …  ▸ 급한이유      │
│   2. …                                         │
│   3. …                                         │
├──────────────────────────────────────────────┤
│  유형별 개수 막대그래프 (작게)                 │
└──────────────────────────────────────────────┘
```
- 화면 최상단 = TOP 3 (사장님이 보려는 것). 나머지는 보조.
- 전체 표는 화면 맨 아래 접이식(expander)으로 숨김.

## 6. 단계별 구현 계획 (가장 작은 것부터)

| 마일스톤 | 내용 | 산출물 | 검증 |
|---|---|---|---|
| **M0** | 날짜만 통일 | `feedback_step1_dates.csv` | 22행 전부 `YYYY-MM-DD` |
