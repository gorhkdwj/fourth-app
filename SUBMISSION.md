# 제출물 — Day 3 카페 VoC 한 화면

## 1. 동작하는 화면
- 로컬 실행: `streamlit run app.py` → http://localhost:8501
- (선택과제로 배포 시 여기에 공개 URL 추가)
- 화면 구성: 유형별 지표 4개 → 🔥 가장 급한 불만 TOP 3 → 유형별 막대 → 전체 표(접이식)

## 2. '가장 급한 불만'을 어떻게 정의했는가 (한 줄)
> **긴급도 = 심각도(S) × 불만강도(I), I = max(별점신호, 텍스트신호)** — 별점을 유일 기준으로 쓰지 않고, 서비스 실패 심각도(SERVQUAL '신뢰성' 최우선)와 고객 불만 강도(별점·텍스트 키워드/감정 중 더 위험한 쪽)를 곱했다. 이는 별점이 아니라 카테고리+키워드+감정강도로 우선순위를 매기는 CS·VoC 업계(Zendesk·IrisAgent·edesk) 방식과 같다. 동점은 금전 손실·환불 요구 건을 우선.

**결과 TOP 3** (모두 신뢰성 실패, 긴급도 12)
1. (12 = S4 × I3) 결제됐으나 포인트 미적립 + 환불 요구 — 금전 손실 (별점1·텍스트 '환불')
2. (12 = S4 × I3) 직원이 주문을 두 번 잘못 받음 — 반복된 주문 정확성 실패 (별점1·텍스트 '두 번')
3. (12 = S4 × I3) 기프티콘 앱 오류 — 결제/사용 차질 (별점없음·텍스트 '오류')

> 별점 없는 채널(전화)도 텍스트신호로 강도가 채워져 id3(앱 오류)이 별점1짜리와 동급 12점으로 진입 — 채널 편향 제거.

## 3. 분류에 쓴 기준 (Few-shot)
유형 ∈ {불만, 요청, 칭찬, 문의}, 감정 ∈ {긍정, 부정, 중립}.
**주 유형 규칙**: 불만족·문제 지적이 있으면 제안이 섞여도 **불만**, 순수 질문은 **문의**, 불만족 없는 순수 제안만 **요청**. 흔들림 방지용 예시:

| 내용 | 유형 | 감정 |
|---|---|---|
| 결제는 됐는데 포인트가 안 쌓였어요. 환불해주세요. | 불만 | 부정 |
| 신메뉴 라떼 너무 달아요. 당도 조절 옵션 있으면 좋겠어요. | 불만 | 부정 |
| 시즌 한정 메뉴 자주 나왔으면 좋겠어요. | 요청 | 긍정 |
| 케이크 퀄리티가 베이커리급이에요. 강력 추천! | 칭찬 | 긍정 |
| 단체 예약도 가능한가요? 10명 정도 인원이에요. | 문의 | 중립 |

별점이 없으면 텍스트신호(키워드/감정강도)가 불만강도를 대신 채운다 — 별점 결측을 상수로 임퓨테이션하지 않음(Chakraborty·Kim·Sudhir 2022, *JMR*: 나이브 임퓨테이션은 판단을 오도).

## 4. 한 줄 회고 — 무엇을 '안 만들기로' 했나
> 경로별·날짜별 추세 그래프, 워드클라우드, 필터 UI를 **안 만들었다**. 사장님 요청은 "지금 가장 급한 불만 하나"였기에, 화면 최상단을 TOP 3에 내주고 전체 표는 접어 두는 '덜어내기'를 택했다. 또 앱이 실시간으로 Claude를 호출하지 않고, 분류는 사전 1회 수행해 정적 CSV로 고정했다(단순·재현 가능).

## 5. 도전과제(선택) 진행

- ✅ **분류 Skill화**: 분류를 데이터(`classifications.csv`)로 분리 + `.claude/skills/feedback-classify` — 새 데이터에 코드 수정 없이 적용
- ✅ **지속 유입 경량**(`ingest.py`): 새 CSV append + 자연키 dedup(멱등) + 증분 분류
- ➕ **추가 기능**: 처리완료 워크큐(긴급도 임계 기반), 실패차원별 분포, 입력검증·회귀검증·원클릭 파이프라인
- ⬜ **Streamlit Cloud 배포**: 보류(GitHub 레포 + 계정 필요 — 사용자 진행)
- ❌ **Supabase 저장**: 소규모 규모엔 과설계로 판단해 제외(trigger 충족 시)
