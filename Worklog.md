## 2. 핵심 판단 과정 — '가장 급한 불만' 정의의 진화

이 과제의 머리 쓸 포인트. 사용자와의 문답으로 기준을 단계적으로 강화했다.

### 2-1. 초안 — 가중치 합산
유형+별점+감정+키워드의 **덧셈** 점수. → 사용자 피드백: *"가중치가 임의적이면 '왜 이 기준?'에 답 못 한다. 학술/실무 근거를 대라."*

### 2-2. 근거 있는 곱셈 모델
**긴급도 = 심각도(S) × 불만강도(I)** 로 전환. (리스크 매트릭스: 영향도×심각도)
- **S**: SERVQUAL 서비스품질 차원 중요도(신뢰성 최우선) 근거 → 신뢰성4·응답성3·유형성2·제품기호1.
- **I**: 별점(CSAT 바텀박스) 기반, 별점 없으면 감정 대체.

### 2-3. 결측 별점(null) 점검 — 사용자 지적으로 발견한 편향
사용자 질문: *"별점 null은 어떻게 점수화되나? 과소/과대평가 아닌가?"*
- **발견**: 별점 없는 '부정'은 강도 상한이 2로, 별점1(3)에 도달 불가. 전화·DM·설문 등 **별점 없는 채널이 구조적으로 과소평가**(채널 편향).
- **근거**: 별점 있는 '부정' 불만의 실제 강도 평균 = 2.6 → 상수 2는 과소.
- **영향**: id3(기프티콘 앱 오류, 무별점)이 8점으로 TOP3 밖. 동일 사안이 별점1이면 12점이었을 것.

### 2-4. 실무 표준 조사 → 모델 B 채택
사용자 요구로 웹에서 업계 방식을 근거와 함께 확인:
- **Zendesk Intelligent Triage**(2024-08 데이터 수집, 2025-05 전 고객 롤아웃): 인텐트·감정(5단계)·언어 자동분류, 부정 감정 시 우선순위 자동 상향 — 별점 아님.
- **IrisAgent**: 우선순위 = 감정+고객가치+비즈니스임팩트+심각도+SLA, "CSAT 기반 스코어링 없음".
- **edesk**: 단어 가중·강도 + 긴급 키워드(refund/chargeback), "별점이 아니라 메시지 내용 기반".
- **Chakraborty·Kim·Sudhir(2022, *Journal of Marketing Research*)**: 결측을 상수로 채우는 나이브 임퓨테이션은 판단을 오도 → 내용 기반 추론 권장.
- **결론**: 별점을 주동인에서 보조로 내리고, 카테고리(심각도)+키워드+감정강도로 산정하는 게 표준. 사용자 승인 → **B 채택**.

### 2-5. 모델 B 확정
**불만강도 I = max(별점신호, 텍스트신호)**
- 별점신호: 1→3, 2→2, 3→1, 4·5→0 (결측 시 미적용) — **별점은 제외되지 않고 한 입력으로 참여**.
- 텍스트신호: 긴급/금전 키워드(환불·결제·포인트·오류·반복)·강한부정 → 3 / 일반부정 2 / 중립 1 / 긍정 0.
- 두 신호 중 **더 위험한 쪽(max)** 채택 → "급한 불만을 놓치지 않기"에 부합.
- 동점: 금전 손실·환불 요구(환불 키워드) 우선 → 최근 날짜순.
- 키워드·가중치는 코드 상수로 분리(조정 가능).

사용자 질문 *"그럼 별점은 아예 제외?"* → 아니오. 예: id6(20분 대기)은 텍스트만이면 강도2지만 별점1이 max로 3을 만들어 6→9로 끌어올림(별점이 살아있음을 입증).

### 2-6. 경계선 분류 보정 — '불만 우선' 규칙
사용자가 대시보드 표에서 id1(당도조절) 긴급도가 None인 것을 발견.
- **원인**: 긴급도는 '불만'에만 부여되는데 id1이 '요청'으로 분류됨(설계상 None, 버그 아님).
- **판단**: 별점2+"너무 달아요"는 불만 성격 → **주 유형 규칙** 도입: 불만족·문제 지적이 있으면 제안이 섞여도 **불만**, 순수 질문은 **문의**, 불만족 없는 순수 제안만 **요청**.
- **적용**: id1·id15 → 요청에서 불만으로. 불만 8→10, 요청 3→1. **TOP3 불변**.
- 부수 개선: 비점수 행의 `None` 표시 → `—`.

## 6. 완성도·확장 구현 (안전 순서로 진행)

필수 완료 후, "안전망부터 → 저위험 → 상태변경 → 자동화 → 배포" 순서로 추가했다.
원칙: 매 단계 `run_pipeline`(끝에 verify) + `AppTest`로 회귀를 즉시 확인.

| 순서 | 작업 | 핵심 |
|---|---|---|
| 안전망 | 검증 레포화(`verify.py`) · 원클릭(`run_pipeline.py`) · 입력검증(`validate.py`) · README | "바꾸고 → 재생성 → 검증" 루프 확보 |
| 기능 | 실패차원별 분포 막대 | 불만이 어느 영역에 집중되나(드릴다운) |
| 상태 | 처리완료 워크큐(`resolved.json`) | 임계 U≥4·최대 3개·되돌리기·빈 상태. 전체 표에 처리 3상태 |
| 자동화 | 분류 Skill화 | 분류를 코드(CLS)→데이터(`classifications.csv`)로, `.claude/skills/feedback-classify` |
| 확장 | 지속 유입 경량(`ingest.py`) | 새 CSV append + 자연키 dedup(멱등) + 증분 분류 안내 |

판단 메모(설계 결정):
- **권장조치 미채택**: 내부 규정을 모르는 처방은 부적절·위험 → 진단까지만, 행동은 사장님께.
- **Supabase 미채택**: 소규모 매장(연 수천 건)엔 CSV로 충분 — DB는 과설계(YAGNI). 배포+상태공유 등 trigger 충족 시에만.
- **워딩**: 대시보드에서 '깨지다/화재' 등 자극적 표현 제거, 중립·전문 워딩.
